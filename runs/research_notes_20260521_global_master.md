## Paper-to-code mapping

- Probabilistic set covering / Benders: our exact objective for one task-key group is already `_group_expected_cost(rows, task_count)`, which encodes success probability and the expected penalty. We do not need scenario SAA for 30-40 tasks; the useful idea is separating a master selection problem from column pricing.
- Set partitioning / column generation: each candidate output group `(task_key, [couriers])` is a column. Master constraints are: each task appears in at most one selected task-key; each courier appears in at most one selected group; uncovered tasks pay `100` each.
- Crowdsourced delivery decomposition: enumerate/score useful route/bundle columns, then solve a restricted master. Current `_ssecr` is closest to this, but only activates in scarce (`G`) and uses hand-tuned pruning.
- Bundling paper: bundles are valuable only when their opportunity cost is below the single-task alternatives they displace. So candidate pruning should use reduced cost / shadow price, not raw row cost.

## Current solver implications

- low501: final picker experiments are exhausted. Improvement must happen before picker by generating different candidate columns, not by choosing among the same columns differently.
- other nine cases: keep existing branch behavior unless a new candidate is strictly better under exact local objective; do not change global picker thresholds without evidence.

## Next experiment design

Build a local-only prototype of a `dual priced restricted master`:
1. Start from incumbent solutions already generated by `solve` branches.
2. Derive task/courier shadow prices from incumbent groups: expensive/risky groups get high task prices; saturated couriers get opportunity penalties.
3. Price single and bundle columns by exact group expected cost minus covered-task prices plus courier pressure.
4. Beam-pack selected columns under task/courier conflicts, preserving uncovered penalty.
5. Add candidate to `E`; final exact objective/picker decides, so non-target cases should not regress.

Guarding rules:
- For scarce401 official cache, never return prototype directly; only include as candidate after cache-protected incumbent exists.
- For low501, run only when `J` classifier is true.
- No official submit until proxy + unit + gate pass and expected value is meaningful.

Result: rejected.

## Experiment D: replace low `_llars` time with deeper `_pces`

Prototype saved: `runs/candidate_low_replace_llars_with_pces_20260521.py`.

Result: rejected for now and reverted.

Evidence:
- Standalone deeper `_pces` can improve some public low incumbents by `13-16` points, while `_llars`/`sbirs` often produce no change.
- Integrated replacement of the final low `_llars` call with `_pces(... top=14, max_k=5, option=120, pairs=80)` did not stably beat the baseline in alternating proxy runs; both baseline and candidate cycled among the same time-sensitive low signatures.
- Runtime on local proxy often hovered around or above 10s, so official risk is not justified without a stronger deterministic gain.

Interpretation:
- Deeper pair exchange is a promising component, but not submit-worthy as a direct time-swap.
- If revisited, make it deterministic and earlier in the low pipeline, or condition it on a low incumbent signature where standalone `_pces` reliably improves.
